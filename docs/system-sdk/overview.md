# System SDK

Rules systems declare:

- `system.manifest.json`
- actor JSON schema
- item JSON schema
- sheet registrations
- dice formula registrations
- character creation templates
- character origin catalogs and create-time origin application where supported by the rules runtime
- character import normalizers
- compendium entries for system-specific items, spells, talents, gear, clues, rituals, and conditions
- condition and compendium action automation for actor quick rolls
- action-use consumption for spell slots, resources, strain, composure, consumable quantities, and system-specific condition side effects
- equipment purchasing and currency updates where supported by the rules runtime
- guided advancement options for leveling or rank progression
- rest and recovery automation for pools, resources, spell slots, and rest-clearing conditions
- encounter threat catalogs and budgeted encounter planning
- actor and monster attack/damage quick rolls where supported by the rules runtime
- optional server hooks

## System Author Checklist

- Provide `system.manifest.json` with a lowercase kebab-case id, semantic version, display name, compatible core range, safe package-relative schema/entrypoint paths, least-privilege permissions, and explicit capabilities.
- Treat manifest metadata as declarative. Registration never grants an external package executable runtime capabilities; the current server admits external packages only with `data-model`, and returns `unsupported_system_capability` for unavailable rules features.
- Keep actor and item data schemas versioned and tolerant of older records; migration-impact notes should describe any field additions, removals, or default changes.
- Ship character templates, compendium coverage, condition metadata, encounter threat catalogs, and action/rest/advancement hooks together so the GM-facing System Registry can show capability coverage before activation.
- Keep legal boundaries explicit: include source/license metadata for reusable rules content and do not bundle proprietary material.
- Add focused tests for character creation, compendium import, action rolls, resource consumption, rest recovery, advancement, encounter planning, and activation/deactivation effects.
- Verify browser activation impact, manifest validation, compendium coverage, and migration posture in the Runtime SDK panel before publishing a system as production-ready.
- Run the [extension package CI](../extension-ci.md) checks for schema compatibility fixtures, rules-action coverage, manifest validation, and activation smoke evidence before sharing a package.

Action metadata now exposes resource costs, condition application, condition clearing, consumable status, and tags; stateful action use can spend Field Prayer to clear Poisoned or spend Second Wind through Guardian Rally to apply Blessed through the same owned-actor or GM-editable roll path.

Tiefling species automation covers Fiendish Legacy and Otherworldly Presence. Tiefling sheets expose `species-tiefling-fiendish-legacy` and `species-tiefling-otherworldly-presence` quick rolls, and `useFreeResource` can spend the matching legacy resource for the selected species spell.

## Public Alpha Smoke Path

The public-alpha one-shot uses `dnd-5e-srd` as the active rules system. A GM can inspect installed systems for the imported demo campaign:

```http
GET /api/v1/campaigns/camp_public_alpha_ember_vault/systems
```

The response should include `dnd-5e-srd` with `active: true` plus the built-in `generic-fantasy`, `stellar-frontiers`, and `mystic-noir` demo systems. Only Generic Fantasy also has an installable SDK example at `plugins/example-system-generic-fantasy`.

The alpha regression also proves that a system module can be installed and audited without granting ambient campaign power:

```http
POST /api/v1/campaigns/camp_public_alpha_ember_vault/systems/generic-fantasy/install
POST /api/v1/campaigns/camp_public_alpha_ember_vault/systems/dnd-5e-srd/install
```

Both activation calls require normal GM `campaign.update` permission. The API writes `system.activate` audit entries with `actorType: "user"`, then restores the demo campaign to `dnd-5e-srd` so the one-shot remains SRD-scoped. Registering a new external manifest is a separate server-admin action that also requires an explicit campaign context and writes `system.install`.
