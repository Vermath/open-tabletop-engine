# v1.0 Release Notes

Date: 2026-05-14
Status: candidate release-note baseline; do not publish v1.0 until `docs/prd-v1-gap-closure.md` release gates are satisfied.

## Summary

v1.0 turns the dogfood virtual tabletop engine into a self-hostable product baseline for unaffiliated GMs. The release keeps the API-first architecture, explicit permissions, proposal-based AI/plugin behavior, SRD-only bundled content boundary, and single-writer SQLite support model.

## Highlights

- First-run owner bootstrap, password auth, invite acceptance, account management, MFA setup, owner workspace creation, organization member add/list/remove with removed-member and bidirectional cross-workspace roster isolation coverage, active workspace switching, active-organization invite create/roster/revoke controls with bidirectional cross-workspace roster isolation coverage, SCIM group-role mapping create/delete controls, and redacted OIDC/SCIM setup posture are available in the browser.
- GMs can create and manage campaigns, scenes, tokens, journals, chat, dice, combat, actors, assets, content imports, AI proposal review, plugins, systems, archive import/export, backup, restore drills, and admin operations without editing source.
- Scene prep includes folder counts, scene search, visible/total summaries, duplication, reorder, activation history, thumbnails, and background assignment.
- Scene Manager compares the selected prep scene with the active player scene across dimensions, tokens, fog, walls, lights, annotations, background status, and actionable drift details.
- GMs can move the currently filtered visible scene set or an explicitly selected scene set into a prep folder, and can bulk-duplicate selected prep scenes.
- Tabletop annotations include layered/grouped pings, rulers, drawings, and grid-snapped circle/line/cone area templates with affected-token summaries, bulk targeting, update history, group nudge/recolor editing, ruler/template endpoint handles, and drawing path-point handles.
- Phone/tablet layouts keep scene filters and token creation controls within the viewport while retaining the lower thumb-zone toolbar and touch Add Token, token drag, dice roll, and chat-send smoke coverage.
- Area templates can record save DC and damage metadata, then roll template damage through the standard dice history.
- Area-template damage can be applied to affected tokens through existing actor HP updates or token damage conditions.
- Area-template saves can be resolved into full or half damage application for affected tokens.
- Area-template damage can account for resistance, immunity, vulnerability, and concentration-check cues.
- D&D Fighter browser workflows verify Second Wind healing/resource consumption and Action Surge roll-only guardrails from the actor sheet.
- Annotation layers can be bulk shown/hidden in the browser without changing persisted campaign annotations.
- Annotation groups can be cleaned up in one browser action while retaining per-annotation delete history.
- Public API coverage now includes route-specific OpenAPI schemas, idempotency policy, rate-limit headers, reusable API-client coverage, and focused security, migration, deployment, performance, and browser smoke gates.
- Release owners can run `pnpm identity:smoke` with real OIDC/SCIM sandbox variables to collect redacted identity-provider readiness evidence against a deployed API or local sandbox app, using `docs/verification/identity-provider-smoke-evidence.md` as the release evidence template.
- Campaign archives support dry-run import, conflict rejection/recovery, conflict skipping, selected collections, dependency warnings, rollback snapshots, and embedded asset validation.
- Chat history export has browser coverage for JSON and NDJSON download and parse validation across public, GM-only, and threaded-reply messages.
- Plugin and system distribution includes marketplace review posture, registry sync from GM and admin surfaces, trust/signature/core-compatibility checks, deployment-matrix trust-policy smoke coverage, permission review, source/status/core marketplace filtering, version install/upgrade/rollback controls, GM system activation, player read-only permission posture, and browser-visible blocked states.
- Production operations include SQLite backup/restore drills, Admin Users reset/session/disable controls, individual Active Sessions revocation, Auth Operations risk-session cleanup and expired-reset pruning, failed email retry, asset-storage migration/cleanup recovery, asset-integrity CDN purge and quarantine recovery, Admin AI stale thread/tool/pending-proposal/approved-proposal recovery controls, backup and restore-drill worker job queueing, a persisted job ledger with worker leasing, Prometheus-style job metrics, alert dry runs plus configured send-alert browser coverage, and deployment runbooks for Docker Compose, systemd, Kubernetes-style hosting, managed app platforms with durable volumes, and preview-only hosted demos.
- Audio/video remains an external-provider handoff in v1, with documented integration patterns for Discord, Zoom, Google Meet, Jitsi, LiveKit, and similar RTC services.
- Public documentation now has a repository-backed site index at `docs/site/index.md`, a versioned public changelog at `CHANGELOG.md`, a local docs-site renderer verified by `pnpm docs:site:check`, and a GitHub Pages workflow at `.github/workflows/docs-site.yml`.

## Compatibility Guarantees

- Supported self-hosted persistence for v1.0 is a single-writer SQLite store plus configured asset storage. The internal JSON-record table layout is not a public integration contract.
- REST, realtime events, `.ottx` campaign archives, `@open-tabletop/api-client`, plugin manifests, and system manifests are the intended extension and integration surfaces.
- Imports accepted by v1.0: `.ottx` archive versions `0.1.0` and `0.2.0`.
- Exports produced by v1.0: `.ottx` archive version `0.2.0`.
- v0.3 SQLite JSON-record stores are opened in place after backup and migration smoke validation.
- Plugin packages are installable only when their trust policy, marketplace review state, requested permissions, and `compatibleCore` range allow the target version.
- AI and plugin runtimes must create proposals or reviewed outputs; they must not mutate campaign state directly.
- Bundled rules/content remain SRD-compatible. Proprietary content scraping, account bypasses, and branded workflow cloning are not supported.

## Known Limitations

- Multi-organization administration has the current v1 owner/admin workspace threshold: owner-scoped workspace defaults, browser-visible owner workspace creation, organization member add/list/role-change/remove workflows, removed-member switch/list denial, active-session workspace switching with scoped campaign lists, active-organization invite create/roster/revoke controls, organization-stamped campaign creation, bidirectional browser roster isolation, and API direct-object active-workspace enforcement.
- Live OIDC/SCIM provider smokes require real IdP sandboxes. `pnpm identity:smoke` provides an opt-in harness and `docs/verification/identity-provider-smoke-evidence.md` defines the evidence template, but skipped output is not final identity-provider evidence and the smoke is not part of the default release gate.
- Rules automation is intentionally generic in v1.0; the browser blocks unsupported roll-only effect application and covers template resistance/concentration notation, combat condition expiry/concentration loss, and representative D&D SRD damage/healing/resource/condition/recovery action effects across character classes and monster actions. Deeper class-feature edge cases remain future hardening.
- Browser E2E coverage is broad and includes two-context player chat/whisper/owned-token movement permission checks, archive reject-conflict and selected-collection import recovery, admin auth/storage/asset/AI/plugin/job/audit recovery controls, plugin/system marketplace filtering/install/upgrade/rollback/read-only/failure states, deleted-asset action disabling plus restore recovery, and touch-enabled mobile/tablet Add Token, token drag, dice, chat, and reload smokes, but it is not exhaustive for every future advanced combat, import/export, asset provider, AI recovery, plugin/system marketplace, admin recovery, or mobile live-play gesture permutation.
- Accessibility validation is currently Playwright label/focus/keyboard coverage plus reduced-motion and contrast checks, including explicit ARIA labels on core tabletop icon controls without relying on `title` as the only name source; the manual assistive-technology pass plan lives at `docs/verification/accessibility-assistive-tech-pass.md`, and completed passes remain required before declaring accessibility complete.
- Layered annotations include layer/group summaries, history, group cleanup/nudge/recolor actions, direct drag handles for moving visible annotations, first-pass endpoint handles for ruler/template resizing, and drawing path-point handles; broader freehand/path editing remains future polish.
- Hosted deployments without durable writable SQLite and asset storage are preview-only unless backed by equivalent persistent volumes and the documented backup/restore process in `docs/deployment/hosted-deployment-recipes.md`.
- Built-in voice/video, recording storage, and provider-side RTC moderation controls are not part of v1; use the documented external handoff boundary.
- Future archive or database versions will need new committed migration fixtures as those versions are released.

## Release Gate

Before publishing v1.0:

```bash
pnpm install --frozen-lockfile
pnpm release:smoke
pnpm v1:completion:audit
pnpm v1:evidence:check
pnpm v1:issues:check
```

If evidence documents are committed after the hosted workflow run, set `OTTE_RELEASE_COMMIT` to the full hosted-run commit SHA before running `pnpm v1:completion:audit` and `pnpm v1:evidence:check`.

The release owner must also confirm the external/manual/hosted evidence gates in `docs/prd-v1-gap-closure.md` are complete or explicitly accepted with owner approval, and that any open P0/P1 issue risk is explicitly accepted by the owner.
The release-owner preflight and rollback checklist lives at `docs/release/v1-release-checklist.md`.
The current completion-audit checklist lives at `docs/verification/v1-gap-closure-completion-audit.md`.
Hosted release-smoke and docs-publication runs should be recorded with `docs/verification/release-workflow-evidence.md`.
External GM validation should be recorded with `docs/verification/external-gm-validation.md`.

When OIDC/SCIM sandbox credentials are available, also run `pnpm identity:smoke` and attach the redacted pass output using `docs/verification/identity-provider-smoke-evidence.md`.
