# OpenTabletop Engine v1.0 Gap Closure PRD

Date: 2026-05-13

## Purpose

OpenTabletop Engine has a strong API-backed v0.3 dogfood slice: campaign data, scenes, tokens, dice, chat, journals, combat records, AI proposals, plugin/system runtimes, import/export, admin operations, and asset delivery all have working code and broad API regression coverage.

The gap to v1.0 is not "add one missing feature." The gap is turning a dense demo cockpit and broad backend into a coherent, reliable product that an unaffiliated GM can adopt, administer, extend, and trust without source-code knowledge or hand-held runbooks.

This PRD defines the product, engineering, UX, operations, and release work required to close that gap completely.

## Current State Summary

### What Works

- A local GM session can load the demo campaign, render a scene, move tokens, upload a map, add fog/walls/lights, roll dice, send chat, edit HP, create a journal note, start combat, preview/apply content imports, run AI proposal/memory actions, run SDK/plugin/system actions, export/report/import campaign data, and switch to a restricted player session.
- API coverage is broad and permission-focused. Auth, sessions, SCIM, OIDC, MFA, assets, admin operations, plugins, systems, AI, import/export, and SRD rules behavior are heavily represented in tests.
- The v0.3 dogfood docs correctly record many limitations and acceptance artifacts.

### What Is Not v1-Ready

- The web app is still a single-file, dense cockpit. Many advanced workflows are compact, operational, or icon-heavy rather than polished product flows.
- The user model now has first-run owner setup, login/register/account/MFA/reset/invite flows, owner-scoped workspace defaults across multiple organization records, browser-visible owner workspace creation, browser-visible organization member listing/addition/role-change/removal for existing users, bearer-session active workspace switching, direct campaign route active-organization enforcement, campaign-invite organization membership attachment, organization membership records with owner/admin add/list/update/remove APIs, organization-stamped campaign creation, and bidirectional browser/API workspace isolation coverage. IdP-specific live provider testing remains an external release-evidence gate.
- Core tabletop workflows now have first v1 coverage in the UI for actor sheets, compendium use, content import, measurement, drawing, advanced targeting, and advanced combat/rules workflows; deeper rules and interaction polish remain future hardening.
- Admin, plugin, system, AI, and asset operations exist mostly as API and dense dashboards, not as discoverable operational workflows.
- Persistence is SQLite-backed and still uses JSON records in one generic `engine_records` table, but the self-hosted v1 support boundary now explicitly treats that as a supported single-writer store rather than a public SQL integration contract. Server admins can inspect schema migrations/indexes/integrity from the Admin UI and run SQLite backup plus restore-drill checks. The `packages/database` relational migration plan remains a placeholder, not a production relational schema.
- The public API client now covers the main reusable public route surface, and `/api/v1/openapi.json` now publishes shared v1 compatibility/auth/error/idempotency/pagination/rate-limit policy. Route-specific OpenAPI schema passes now cover high-traffic auth, auth-management, OIDC JSON callback/start, SCIM, invite, campaign, scene, scene-tool, token, chat, dice, asset, actor, item, journal, combat, proposal, encounter, AI, plugin, system, content-import, campaign import/export, chat export, dogfood-report, admin auth/user/session/email/audit, admin AI/plugin/system/rendering/asset operations, asset-blob delivery, OIDC browser redirect, and admin storage-operation routes. The generated contract no longer has generic `200` responses without response content.
- A durable browser E2E suite now covers critical smoke paths and the first advanced-flow threshold; final v1 acceptance still depends on the external/manual identity, assistive-technology, and docs-publication evidence gates.
- Mobile/tablet now opens into the live tabletop workspace before the management rail, and the earlier observed player reload path that could land on an empty campaign header now has a snapshot-ready guard and reload coverage. The phone/tablet smoke now runs in touch-enabled contexts, verifies topbar controls fit the viewport, taps Add Token through to an API-visible scene token, touch-drags that token until the API coordinates change, and then cleans it up. It is still not a fully polished touch-first tabletop experience.

## v1.0 Release Definition

v1.0 is complete when a new self-hosting GM can:

1. Create or join an organization, configure authentication, and invite players without demo shortcuts.
2. Create a campaign, choose a rules system, import or build content, upload maps/assets, and configure scenes.
3. Run a live multi-user session with reliable realtime, explicit permissions, usable GM/player UIs, and recoverable state.
4. Use AI only through reviewable proposals with clear audit trails and failure recovery.
5. Install and manage plugins/systems through signed, permissioned, reviewable workflows.
6. Export, import, backup, restore, and upgrade campaign data with documented compatibility guarantees.
7. Operate the server with admin dashboards, alerts, audit exports, and storage/job health.
8. Validate the release through automated API, browser, security, accessibility, performance, migration, and upgrade tests.

## Product Requirements

### Epic 1: First-Run, Auth, and Organization Setup

**Goal:** Replace demo session switching as the primary user model.

Requirements:

- First-run owner bootstrap flow creates the initial server admin and organization.
- Login, register, logout, password reset, password change, MFA setup/recovery, and account settings are all web-client flows.
- OIDC and SCIM configuration have admin setup screens with validation and redacted test connection actions.
- Invite acceptance creates a real account or links an existing account.
- Demo mode remains available only as an explicit development/demo toggle.

Acceptance criteria:

- A clean deployment with no users routes to owner setup.
- A player can accept an invite in a private browser session and land directly in the campaign.
- A server admin can configure MFA/OIDC/SCIM without editing source.
- Browser E2E covers owner setup, invite, login/logout, reset, MFA, and OIDC-disabled fallback.

### Epic 2: Campaign and Scene Management

**Goal:** Let a GM build and manage a campaign without hardcoded demo flows.

Requirements:

- Campaign create/edit/archive/delete screens.
- Campaign setup wizard: name, description, rules system, visibility, default permissions, players, and initial scene.
- Scene manager: create/edit/delete/duplicate/reorder/activate scenes.
- Scene editor: dimensions, grid, background asset, fog state, lighting defaults, wall mode, and player visibility.
- Undo/audit affordances for destructive campaign and scene changes.

Acceptance criteria:

- GM creates a campaign from scratch, uploads a map, creates two scenes, activates one for players, and exports the campaign.
- Browser E2E creates a campaign from the setup wizard with a custom first scene, starter player invite, and onboarding journal handout.
- Player sees only active/public scenes and no GM-only prep state.
- Campaign import/export round-trip preserves scene metadata and assets.

### Epic 3: Tabletop Runtime UX

**Goal:** Make the live tabletop usable for real play.

Requirements:

- Token inspector for name, actor link, size, position, image, disposition, visibility, ownership, conditions, auras, vision, and notes.
- Drag/drop token placement from actor/asset lists.
- Measurement ruler, templates, ping, drawing/annotation, and clear/select tools.
- Fog/wall/light workflows use in-app dialogs and direct manipulation, not prompt/alert.
- Chat supports full history, scrollback, filters, whispers, emotes, GM messages, OOC, moderation, and export.
- Dice supports roll details, roll privacy, macros, saved formulas, advantage/disadvantage, and system-aware rolls.

Acceptance criteria:

- GM can prep a map and run movement, fog, light, measurements, chat, and dice without native browser dialogs.
- Player can move owned tokens only, whisper the GM, roll privately/publicly, and inspect only permitted actor/journal data.
   - Browser E2E validates GM/player realtime and permission boundaries in two independent browser contexts, including accepted-player active-scene visibility with a hidden GM prep scene excluded, public and GM-only player dice rolls, owned-token movement, unowned-token movement denial, owned-character sheet action use with visible Action Surge resource consumption, GM-visible realtime chat, player-to-GM whisper, and combat ready/delayed state, conditions, death saves, resource tracking, and actor-sheet sync from combat edits.

### Epic 4: Actors, Character Sheets, Rules, and Combat

**Goal:** Turn rule runtime depth into playable user workflows.

Requirements:

- Rules-system selection and active-system switching are explicit, validated flows.
- Character sheet UI for the primary v1 system, with tabs for stats, resources, actions, spells, inventory, conditions, notes, and permissions.
- Compendium browser with search/filter, source/license metadata, add-to-actor, add-to-scene, and add-to-journal actions.
- Character builder/importer with validation, conflict handling, and legal content boundaries.
- Combat tracker supports initiative editing, turn advancement, round advancement, conditions, HP/resource updates, hidden enemies, defeated state, actor-sheet sync for combat conditions/death saves/resource-used state, end combat, and audit history.
- Target selection connects token targeting to system actions and AI/tool proposals.

Acceptance criteria:

- GM can create/import a party, add monsters, start combat, run at least three rounds, apply damage/healing/conditions, and end combat. Browser coverage now creates a D&D SRD monster from the SDK panel, places it as a scene token, starts a multi-combatant combat, edits HP down/up, applies conditions/resource/death-save state, advances to round 3, and ends combat.
- Player can use an owned character sheet action and see resource consumption; browser coverage now links a player-owned level-2 fighter to an owned token, spends Action Surge from the accepted-player sheet UI, and verifies the resource changes from `1/1` to `0/1`.
- Unsupported rules-system actions show clear unsupported-state messages instead of silent no-ops; the browser Actor Actions smoke now verifies a roll-only Fighter Action Surge action is labelled and disabled while Apply Effect is enabled.

### Epic 5: Content, Assets, Import/Export, and Backups

**Goal:** Make campaign portability and content safety complete.

Requirements:

- Asset library with thumbnails, folders/tags, search, quotas, lifecycle status, restore/delete, and CDN/delivery diagnostics.
- Archive export wizard with scope, version, redaction, and compatibility notes.

- Backup/restore UI and CLI with scheduled backups and restore drills.
- Export compatibility policy for `0.1.0`, `0.2.0`, and v1 archive versions.

Acceptance criteria:

- Failed import leaves no partial state unless explicitly applied.
- Report bundle is downloadable from a clear issue-reporting flow.

### Epic 6: AI Proposal Productization

Requirements:

Acceptance criteria:
