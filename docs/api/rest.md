# REST API

The API is served from `apps/api` and exposes:

Authenticated endpoints require a bearer session token. Create one with `POST /api/v1/auth/login` or `POST /api/v1/auth/register`, then send `Authorization: Bearer <token>` on REST requests. Password-backed users authenticate by email and password. The seeded local users are `usr_demo_gm` and `usr_demo_player`; they remain passwordless for local test compatibility. The legacy `x-user-id` header is disabled in production and is available only in tests or non-production compatibility mode with `OTTE_ALLOW_LEGACY_USER_HEADER=true`. Asset blob and realtime URLs accept `sessionToken=<token>` for contexts that cannot set an `Authorization` header.

OIDC SSO is enabled when `OTTE_OIDC_ISSUER` and `OTTE_OIDC_CLIENT_ID` are set. The API discovers the provider metadata, starts an authorization-code flow with PKCE/state/nonce, exchanges the callback code, reads the OIDC userinfo endpoint, links an existing user by normalized email when possible, and issues the same opaque `ots_` bearer session token used by password login. Production return origins should be set with `OTTE_WEB_ORIGIN` or `OTTE_OIDC_ALLOWED_RETURN_ORIGINS`; localhost return origins are allowed for development.

Plugin runtime endpoints are campaign-scoped. `GET /api/v1/plugins` returns manifest-loaded local and registry-synced packages with source metadata, sandbox mode, manifest/server-entrypoint checksums, version distribution metadata, and trust status. `POST /api/v1/plugins` registers an additional package by `packagePath` under the configured plugin root. `POST /api/v1/plugins/registry/sync` fetches one configured remote registry from `OTTE_PLUGIN_REGISTRY_URLS`, or every configured registry when no `registryUrl` is supplied, and mirrors JSON plugin package files into the configured plugin root before loading them through the same manifest, sandbox, versioning, and trust checks as local packages. Server admins can run the same configured-registry sync without campaign context through `POST /api/v1/admin/plugins/registry/sync`; both paths are allowlist-bound and audited. Registry sync refuses to overwrite existing local package directories that do not already carry matching registry provenance. Registry catalog entries use `packageId`, `packageUrl` or `downloadUrl`, and optional `sha256:` `checksum`; downloaded package documents contain a `files` object with `plugin.manifest.json`, server/client entrypoints, and optional `plugin.signature.json`. Server admins can review every package version/checksum through `GET /api/v1/admin/plugins/reviews` and set `pending`, `approved`, or `rejected` with `PATCH /api/v1/admin/plugins/reviews/{reviewKey}`; rejected packages are blocked, and `OTTE_PLUGIN_REVIEW_POLICY=require_approved` blocks install and execution until the matching package review is approved. `GET /api/v1/admin/plugins/operations` returns the trust/review policy, top-level action-required state, machine-readable action reasons, a prioritized remediation queue with severity/action/sample context, package load errors, installed campaign grants, missing runtime packages, blocked installs, permission drift, core compatibility drift against the current server core version, storage counts, storage operation rollups, install/change history counts, command operations counts, failed command counts, command failure plugin/reason rollups, recent storage/install/change/rollback, recent command and failure audit summaries, duplicate package-version inventory warnings, registry health posture, and review backlog/service-level details such as approval coverage, pending/rejected/blocked review counts, oldest pending review age, source counts, review action reasons, remediation guidance, and bounded pending/approved/rejected/blocked package samples for production operations. Registry operations report configured registry URLs, invalid or production-insecure `OTTE_PLUGIN_REGISTRY_URLS` posture and malformed `OTTE_PLUGIN_REGISTRY_STALE_SECONDS` or `OTTE_PLUGIN_REGISTRY_TIMEOUT_MS` numeric config using only redacted environment-variable names and counts, never-synced registries, latest sync errors from audit logs, sync age/staleness against `OTTE_PLUGIN_REGISTRY_STALE_SECONDS` (default seven days), registry-sourced package counts, stale configured-registry samples, and packages synced from registries no longer configured on the server. Security posture reports trust policy, trust-key count, production unsigned-package exposure, trusted-mode-without-keys exposure, VM-sandbox coverage, unsigned/untrusted package counts, and bounded package samples without exposing trust-key material. Action reasons are `load_errors`, `blocked_installs`, `missing_installed_packages`, `permission_drift`, `core_compatibility_drift`, `review_backlog`, `plugin_command_failures`, `duplicate_plugin_packages`, `plugin_trust_policy_allows_unsigned_in_production`, `plugin_trust_keys_missing`, `plugin_registry_url_config_invalid`, `plugin_registry_url_insecure`, `plugin_registry_numeric_config_invalid`, `configured_registry_unsynced`, `registry_sync_errors`, `stale_plugin_registry_sync`, and `unconfigured_registry_packages`. Plugin storage audit rows roll into `storageOperations` with direct set/delete counts, command-mutation counts, set/delete mutation totals, per-plugin and per-campaign rollups, and bounded recent rows containing campaign/plugin ids, actor type/id, operation, key, value size, delete result, and timestamp, without exposing stored JSON values. Plugin install audit rows roll into `installOperations` with install, upgrade, rollback, and permission-review counts plus bounded recent rows containing campaign/plugin ids, actor user id, package id, version, sandbox, permission counts, and timestamp, without exposing plugin code or command payloads. Failed VM command execution and invalid storage mutations append redacted `plugin.chatCommand.failed` audit rows; those rows drive the `investigate_plugin_command_failures` remediation item with bounded plugin, command, reason, and message samples. Duplicate package-version inventory drives `dedupe_plugin_package_versions` remediation with bounded plugin, version, package id, and source-type samples so shadowed plugin packages are visible before install or command resolution becomes ambiguous. Near-limit plugin storage drives `trim_plugin_storage_pressure` remediation with campaign/plugin/key, value size, max-value limit, last writer type, and update timestamp samples so operators can distinguish plugin-written cache pressure from user-authored configuration. The browser Admin tab can trigger `POST /api/v1/admin/plugins/registry/sync` for configured registries, refresh plugin operations/review posture, and display each configured registry's sync status, synced package count, last sync time, sync age/staleness, latest sync errors, top remediation actions, recent storage operation rows, recent install/version-change rows, recent command activity, recent command failures, review remediation, pending/approved/rejected/blocked review samples, duplicate package-version rows, and storage-mutating command counts alongside unconfigured registry-package drift. Installing a plugin creates or updates a `permissionGrants` record for `subjectType: "plugin"`; callers may grant all requested permissions or a subset for review, and may request a specific package `version`. Campaign plugin storage is a JSON key/value API under `/api/v1/campaigns/{campaignId}/plugins/{pluginId}/storage`; human callers need `plugin.configure`, and the installed plugin grant must include `plugin.configure`. Storage keys are bounded to 80 URL-safe characters, values are JSON-serializable and capped at 16 KiB, storage entries persist in campaign archives/SQLite, and writes/deletes are audited. VM command handlers receive a snapshot of their plugin storage when granted `plugin.configure` and can return bounded storage mutations alongside the chat result. When `OTTE_PLUGIN_TRUST_POLICY=require_trusted`, unsigned or tampered plugin packages are visible for review but blocked from installation and command execution unless their `plugin.signature.json` verifies against `OTTE_PLUGIN_TRUST_KEYS`. Command execution checks the human caller's permission, the current trust policy, marketplace review state, core compatibility posture, and the plugin grant before the plugin can post chat, read token context, or mutate plugin storage, then runs the manifest-declared server command in the VM sandbox.

Plugin operations also include a `securityPosture` block that counts VM-sandboxed versus manifest-only packages, command-capable packages, unsigned/untrusted/trust-blocked package inventory, and any command-capable package that is not VM-backed. Those sandbox/trust findings also drive `review_plugin_security_posture` remediation with bounded package samples and issue labels, so production operators can triage package-security exposure from the top-level queue before enabling broader third-party distribution. The Admin tab surfaces this posture beside marketplace and registry operations.

Server admins can inspect rules ecosystem posture with `GET /api/v1/admin/systems/operations`. The audited response summarizes installed system manifests, active campaign counts, actor/item usage by system, template/compendium/condition/threat coverage, artifact-derived origin, monster-creation, and equipment-purchase evidence, manifest completeness, content issue counts, production-readiness posture, action-required state, and machine-readable action reasons. Action reasons are `system_content_issues`, `non_primary_runtime_in_use`, and `non_primary_runtime_recent_activity`. The D&D SRD runtime is reported as the primary production-ready system; active campaigns, actors, or items on non-primary demo runtimes are grouped under `productionReadiness.systemsNeedingProductionDepth` and make operations actionable so operators can plan migration or runtime hardening. Rules actor rolls, rests, and advancement append redacted `system.actor.roll`, `system.actor.rest`, and `system.actor.advance` audit logs with campaign, actor, actor type, system id, action id/rest type/advancement option, and mutation booleans, but not formulas, actor data, item data, or roll totals. The `activityOperations` block rolls those audit rows into total activity, non-primary activity, per-action/system counts, systems with recent activity, and bounded recent activity samples. Each system also reports a `productionCapability` matrix with supported/missing capabilities, evidence counts, bounded evidence samples, coverage rate, and remediation for character templates, compendium, conditions, encounter threats, origin building, monster creation, equipment purchase, import, advancement, rest recovery, and encounter planning. Ranked production gap rollups include severity, an operator-readable message, remediation guidance, and affected-system samples for non-primary runtime usage, missing origin builders, monster creation, equipment purchase, encounter planning, compendium support, and manifest/content gaps. The `promotionBlockers` list expands those per-system gaps into concrete blocker messages and remediation for each non-production runtime so operators can see what prevents Generic Fantasy, Stellar Frontiers, Mystic Noir, or future systems from being promoted to primary depth. A top-level `remediationQueue` mirrors those ranked rules gaps with severity, action text, affected counts, and bounded runtime usage samples, including recent non-primary rules activity, so operators can triage rules hardening without decoding raw gap ids. The browser Admin tab shows the same rules operations summary with action reasons, production-readiness counts, non-primary campaign/actor/item usage, recent rules activity counts, recent demo-runtime activity rows, content issue counts, top remediation queue rows, promotion blockers, gap severity/remediation guidance, the concrete runtime ids that need production depth, their per-system usage pressure, per-system capability coverage, missing capability labels, capability evidence samples, and per-system template/compendium/condition/threat coverage.


The default `upsert` mode replaces records with matching ids and inserts missing records, which supports round-tripping a campaign into a fresh instance or refreshing an existing imported campaign. Use `reject_conflicts` to return `409` when imported ids already exist.

Archives include uploaded asset files in a top-level `files` array, regardless of whether the active backend is local disk or S3-compatible object storage. Each entry stores base64 data plus size and `sha256` checksum metadata. Import validates the asset id, size, and checksum before restoring the file through the configured asset storage provider, so uploaded map backgrounds can round-trip into a fresh instance. Asset storage operations also separate unscanned uploads from managed assets with persisted medium/high trust-scan findings; those assets appear under `asset_trust_warnings` and `review_asset_trust_warnings` with bounded asset ids, scanner names, finding counts, severities, and finding codes but no scanner URLs, tokens, object bytes, or checksums.
