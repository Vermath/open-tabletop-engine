# REST API

The API is served from `apps/api` and exposes:

Authenticated endpoints require a bearer session token. Create one with `POST /api/v1/auth/login` or `POST /api/v1/auth/register`, then send `Authorization: Bearer <token>` on REST requests. Password-backed users authenticate by email and password. The seeded local users are `usr_demo_gm` and `usr_demo_player`; they remain passwordless for local test compatibility. The legacy `x-user-id` header and `userId` login body are disabled in production and are available only in tests or non-production compatibility mode with `OTTE_ALLOW_LEGACY_USER_HEADER=true`. Browser media should use signed asset delivery URLs, and browser realtime clients should pass bearer sessions through a websocket subprotocol such as `otte.auth.<token>` instead of putting tokens in URLs. Query-string `sessionToken` authentication is deprecated compatibility only: its value remains redacted from request logs, responses carry a non-secret compatibility diagnostic, and operators can set `OTTE_URL_SESSION_TOKEN_MODE=disabled` after legacy clients migrate. Unknown explicit mode values fail closed. Normal first-party clients do not use this compatibility path.

Fresh empty deployments expose first-run owner setup at `GET|POST /api/v1/auth/bootstrap`. `GET` returns whether setup is required; `POST` is accepted only while there are no users, creates the first password-backed owner/server-admin user, creates a private starter campaign and active scene, and returns the same bearer-session shape as login plus the created campaign and scene. Production stores start empty by default unless `OTTE_DEMO_SEED=true`; non-production local stores keep demo seeding unless `OTTE_DEMO_SEED=false`.

OIDC SSO is enabled when `OTTE_OIDC_ISSUER` and `OTTE_OIDC_CLIENT_ID` are set. The API discovers the provider metadata, starts an authorization-code flow with PKCE/state/nonce, exchanges the callback code, and reads the OIDC UserInfo endpoint. An existing `(issuer, sub)` identity is always resolved first. A new identity may link to an existing normalized-email user only when UserInfo returns both an email and the literal JSON boolean `email_verified: true`; a missing, false, or string-valued verification claim instead creates an isolated OIDC user without a primary email. This assumes the configured issuer controls and truthfully emits its UserInfo claims. The callback then issues the same opaque `ots_` bearer session token used by password login. Production return origins should be set with `OTTE_WEB_ORIGIN` or `OTTE_OIDC_ALLOWED_RETURN_ORIGINS`; localhost return origins are allowed for development.

Server admins can run redacted enterprise-auth readiness checks with `POST /api/v1/admin/auth/test-connection` and `{ "provider": "oidc" }` or `{ "provider": "scim" }`. OIDC tests validate required config and provider discovery without returning issuer, client, or secret values. SCIM tests validate bearer-token readiness and endpoint posture without echoing the bearer token.

`GET /api/v1/openapi.json` serves the current OpenAPI 3.1 contract seed from `@open-tabletop/api-contracts`. It now publishes shared v1 policy extensions for route compatibility, bearer auth, the standard error envelope, idempotency-key expectations for mutating routes, cursor/limit pagination, filtering/sorting parameter names, and runtime rate-limit semantics. The route-specific request/response schemas are still being hardened toward v1, so the spec is the compatibility and policy source of truth today rather than a complete generated conformance artifact.

Versioning policy: public routes live under `/api/v1`; v1 additions should be backward-compatible, and incompatible request/response semantics require a new `/api/v{n}` base path. Error responses should keep a stable machine-readable `error` field and may include `message`, `code`, `details`, and `requestId`. Mutating clients should send `Idempotency-Key` per logical user action; the runtime persists successful JSON mutation responses in the engine store and replays matching method/path/user/body requests with `Idempotency-Replayed: true`, while reusing a key for a different request returns `409`. List-style `GET` routes should converge on `limit`, `cursor`, `filter`, and `sort`. The API runtime enforces a fixed-window per-route caller rate limit when `OTTE_RATE_LIMIT_ENABLED=true` and enables it by default in production; throttled requests return `429` with `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset`. Tune the limiter with `OTTE_RATE_LIMIT_WINDOW_SECONDS` and `OTTE_RATE_LIMIT_MAX_REQUESTS`.

The campaign compendium is a server-filtered catalog and does not require an actor or selected map token. `GET /api/v1/campaigns/{campaignId}/systems/{systemId}/compendium` accepts `q` for case-insensitive text search, `type` for one entry type, or comma-separated `types` for multiple types. Its response includes the matching `entries`, normalized `filters`, and a `provenanceSummary` with total/filtered counts, type counts, and source/license rollups. Every returned entry carries `sourceKind`, `sourceName`, `sourceVersion`, `contentVersion`, `systemId`, `systemVersion`, `rulesVersion`, and a license object with its usage classification and optional URL/attribution. The D&D catalog is the SRD 5.2.1 catalog under CC BY 4.0; bundled example systems are marked as open MIT content. The endpoint does not expose proprietary D&D content.

Campaign-authored compendium entries are actorless `Item` records with both `data.compendiumId` and a complete `data.compendiumProvenance` object whose `sourceKind` is `user`. Because items do not yet have their own visibility policy, those entries are returned only to callers with `campaign.update`; ordinary campaign readers receive only the bundled catalog. Missing, malformed, or cross-system provenance is excluded instead of being guessed. Item/condition imports retain provenance on the created actor state, and campaign archive export/import preserves that data unchanged.

Compendium actor writes require `actor.update`, an `Idempotency-Key` header, and the actor's exact `expectedUpdatedAt` revision in the JSON body. Retrying the same key replays the original response. A changed actor returns the standard `409 stale_write` response. An entry already present returns `409` with `code: "compendium_conflict"`, the requested/existing content versions, and only the allowed explicit choices; the server does not silently stack, replace, or charge currency. Import conflicts allow `keep_existing` or `replace_existing`. D&D equipment purchases allow `keep_existing` or `merge_existing` for an exact-version duplicate, and `keep_existing` or `replace_existing` for a version conflict. `merge_existing` adds the requested quantity and cost to the existing item while preserving its item id. `replace_existing` also preserves the item id but resets its catalog content, quantity, and recorded purchase cost to the requested version and purchase. `keep_existing` performs no state or currency mutation.

System condition writes require explicit actor edit permission and optimistic actor revisions. Applying a condition requires an `Idempotency-Key` header plus `expectedUpdatedAt` in the JSON body; removing one requires `expectedUpdatedAt` as a query parameter and also supports a stable `Idempotency-Key` for retries. A changed actor returns the standard `409 stale_write` envelope, successful retries replay the original response without incrementing Exhaustion or duplicating effects, and apply/remove operations append bounded actor audit rows containing only the condition id and revision metadata.

`GET /api/v1/campaigns/{campaignId}/systems/{systemId}/actors/{actorId}/calculation-explanation` is a read-only D&D 5.5e SRD calculation trace. It requires `actor.read` plus access to the actor's private data through `actor.readPrivate`, ownership, or an actor-specific grant. The response reports the server-authoritative result and ordered source terms for ability scores and modifiers, proficiency, Armor Class, hit points, initiative, saves, skills, passive Perception, speed, spellcasting when applicable, and visible action formulas. Every field has explicit `manual`, `override`, `unsupported`, and `ambiguous` flags with reasons; terms identify actor, system, class, feature, item, condition, override, or manual provenance. The formatter consumes existing rules helpers and does not introduce a second formula language or mutate actor state.

`GET /api/v1/campaigns/{campaignId}/compatibility` requires `campaign.update` and returns a deterministic, read-only campaign compatibility report for GMs and editors. It covers the platform and archive versions, every referenced rules system and core-version range, D&D actor/item schema validation, missing or cross-system references, compendium provenance/version drift, and calculation fields that need manual review. Blocking issues sort before warnings and include an explicit next step. `validation.repairPreview` lists only deterministic repair candidates, including exact before/after values and inverse patches. It never applies those candidates, guesses intent-sensitive rules, replaces compendium entries, drops unknown/homebrew fields, or changes campaign state. Actor-private repair details are also available from `GET /api/v1/campaigns/{campaignId}/systems/dnd-5e-srd/actors/{actorId}/rules-validation` to users with private actor access.

D&D controlled-creature routes model summons, transformations, and persistent companions as typed actor lifecycle metadata. Preparation is side-effect-free and returns a preview token, human-review warnings, and optimistic revisions for every actor, item, token, combat, scene, and encounter root that confirmation may change. Confirmation, command, dismissal/expiry, transformation reversion, and concentration cleanup require `Idempotency-Key`; stale roots return `409` without partial mutation. Concentration cleanup ends the linked effect and every creature in its group atomically. Transformations preserve an exact actor/item/combat snapshot and require explicit hit-point and equipment carryover choices. Persistent companions remain archiveable actors after dismissal, while temporary summons are removed with their linked tokens. D&D post-creation spell preparation is a separate reviewed workflow. `POST /api/v1/campaigns/{campaignId}/systems/dnd-5e-srd/actors/{actorId}/spell-preparation/preview` accepts actor-owned spell item ids, an explicit `long-rest` or `class-level` timing acknowledgement, and exact actor/spell-item revisions. The server derives capacity, always-prepared exclusions, matching class provenance, and Wizard spellbook membership from stored state and returns blockers plus exact prepare/unprepare changes. `POST .../spell-preparation/apply` accepts only the prepared preview key and the same revisions, recomputes the plan from current server state, and atomically updates actor and item preparation state with an audit record. Both calls require actor-update authority and `Idempotency-Key`; replay is safe and stale state returns `409` before mutation. Guided level-one class spells are automated. Later-level acquisition, incomplete legacy/import provenance, and homebrew whose legality cannot be proven return manual blockers rather than guessed preparation rights. Generic item PATCH cannot change `prepared` on provenance-managed D&D class spells.

D&D character review is optional for every legacy and newly created campaign until a DM or campaign editor explicitly sets `characterReviewPolicy.mode` to `required`. `GET /dnd/character-reviews` returns only characters whose private actor data the caller may read, with current validation, a build fingerprint, portable submission/decision evidence, and the exact actor/item revisions needed to submit safely. Owners or actor editors submit with an `Idempotency-Key`; DMs decide with `campaign.update`. Requesting changes requires a reason. Validation errors block approval unless the DM explicitly sets `overrideValidation: true` and records a reason. All writes are revision-checked, audited, replay-safe, and stored in campaign/actor data so archive export/import preserves the policy, approval, rejection reason, and manual exception.

The fingerprint covers build-defining actor and owned-item content but excludes volatile play state such as current hit points, spent resources, conditions, quantities, preparation, equipment, and attunement. Ordinary damage, rests, and resource use therefore do not revoke approval; changes to level, maximum values, abilities, item definitions, ownership, or other build data do. Guided level-one characters persist their `level-one-srd` template and exact creator options, and review replays the strict creator legality validator so tampered or forged choices appear as blocking `creation.*` errors. Legacy, imported, and homebrew actors without that explicit provenance are not assigned reconstructed creator-choice errors.

When the policy is required, a current matching approval is checked before placing a linked D&D character token or adding the character to combat. The API returns `409 character_approval_required` before mutation for unsubmitted, rejected, or stale characters. Optional campaigns retain the previous token/combat behavior. The Compendium inspector exposes the same permission-aware queue: players can submit owned characters, while DMs can set policy, inspect field-level issues, approve, request changes, or record a reasoned validation exception. Rules-system registration is a privileged server operation, distinct from campaign activation. `POST /api/v1/systems/install` requires an `Idempotency-Key`, an explicit `campaignId`, that campaign's `campaign.update` permission, and server-admin authority. Its `manifest` must use a lowercase stable id, semantic version, compatible core range, safe package-relative entrypoint/schema paths, the restricted system permission set, and declared capabilities. Duplicate ids, names, and entrypoints are rejected. Approved external manifests are persisted in the engine state and SQLite record model and produce a `system.install` user audit row; campaign exports record referenced systems in `manifest.systemRequirements` without importing an archive as trusted executable code. This server currently accepts external systems as `data-model` packages only. Campaign owners can activate any installed system with `POST /api/v1/campaigns/{campaignId}/systems/{systemId}/install`, which writes a `system.activate` audit row. Calling a sheet, action, rest, compendium, advancement, or encounter endpoint for a data-driven system returns `422 unsupported_system_capability` instead of running Generic Fantasy rules.

Campaign and scene duplication use dedicated server commands rather than browser export/import or repeated creates. `POST /api/v1/campaigns/{campaignId}/duplicate` copies the full campaign graph once. `POST /api/v1/campaigns/{campaignId}/scene-duplications` accepts one stable `operationId`, the exact campaign revision, and one or more exact source-scene revisions. `dryRun: true` returns deterministic target ids, copy counts, shared assets/worlds, and skipped historical or partial references without changing campaign state. Commit repeats the same request with `dryRun: false`; it copies selected scenes, their tokens, token actors and items, active calculation overrides, and fully-contained encounters in one durable state replacement, remaps internal references, leaves shared resources shared, bumps the campaign once, broadcasts created records, and replays the same response for the same `Idempotency-Key`. Any validation, permission, stale-revision, or persistence failure leaves the entire graph unchanged.

Archiving/restoring a campaign is non-destructive: `POST /api/v1/campaigns/{campaignId}/archive` sets `archivedAt`/`archivedByUserId`, `POST /api/v1/campaigns/{campaignId}/restore` clears that archived state and sets `restoredAt`/`restoredByUserId`, and both routes require `campaign.update`, broadcast `campaign.updated`, and write redacted audit rows. Deleting a scene removes the scene, its tokens, scene-scoped chat, source fog presets, and deleted-token references from encounters and combats, then writes a redacted `scene.delete` audit row with scene metadata and aggregate child counts. Scene annotations support `ping`, `ruler`, `template`, and `drawing`; ping/ruler creation requires `scene.read`, template/drawing creation and annotation deletion require `scene.update`, and each create/delete appends a redacted `scene.annotation.*` audit row. Operational retention is preservation-first and server-admin-only. `GET /api/v1/admin/retention/operations` measures terminal delivered-email, delivered-webhook, and maintenance-job ledgers without making them eligible automatically. `POST /api/v1/admin/retention/prune` requires an `Idempotency-Key`, one or more supported record classes, an explicit age, and a bounded batch. Server-admin jobs are persisted in the engine state as an operational ledger. Every job mutation requires an `Idempotency-Key`; job update, heartbeat, retry, and cancellation also require the exact displayed `expectedUpdatedAt`. Human server-admin sessions retain the full job-management surface: list, queue, inspect, retry, cancel, operations, metrics, and alerts. A dedicated worker principal using `Authorization: Worker <token>`, `x-otte-worker-id`, and (after leasing) `x-otte-worker-job-id` may only lease, heartbeat, settle its own active unexpired lease, and invoke the exact allowlisted method/path/body derived from that job payload. Worker tokens are not user sessions and are denied on ordinary account, plugin-admin, server-admin, and campaign-read APIs. `POST /api/v1/admin/jobs/lease` requires a `leaseRequestId` exactly equal to its `Idempotency-Key`, returns a monotonic `leaseRevision`, and returns the raw execution payload only after a named configured token hash matches the presented stable worker id. Heartbeat and settlement are bound to that exact lease revision. General job responses continue to redact archives, AI source text/transcripts, token/password-like fields, and sensitive log details. A dispatch claim is recorded once per lease to prevent replay, and expired, cancelled, mismatched, cross-job, or cross-campaign execution is rejected. `POST /api/v1/admin/jobs/alerts` requires a stable `deliveryId`; that same identity is sent in downstream idempotency headers so an operator retry cannot duplicate an alert. The worker CLI can run one leased job with `OTTE_WORKER_LEASE_ONCE=true` or continuously poll with `OTTE_WORKER_LEASE_POLL=true`, heartbeat execution, dispatch through the existing API bridge, and settle the ledger succeeded or failed. Configure plaintext `OTTE_WORKER_TOKEN` only on the worker and named SHA-256 entries in `OTTE_WORKER_TOKEN_HASHES` only on the API; repeated hashes for one stable id support overlap during rotation. `OTTE_WORKER_SESSION_TOKEN` and `OTTE_SESSION_TOKEN` are deprecated, require explicit non-production compatibility, and are hard-refused in production.

The SQLite JSON-record model is the supported single-writer self-hosting store for v1 hardening work. Its tables are internal implementation details, not a public SQL integration contract. External integrations should use REST, realtime, SDKs, and `.ottx` campaign archives; SQLite backup files are operational recovery artifacts for the same deployment family. File and memory stores remain development/test stores and report unsupported storage operations.

Proposal approval is a two-step flow: `POST /api/v1/proposals/{proposalId}/approve` marks a pending proposal as approved, and `POST /api/v1/proposals/{proposalId}/apply` mutates campaign state only after approval. Applied proposals capture domain-aware inverse changes; `POST /api/v1/proposals/{proposalId}/revert` applies those inverse changes once and records who reverted the proposal. Proposal responses include optional `history` entries for create, approve, reject, apply, revert, and server-admin stale rejection cleanup; each entry records the lifecycle timestamp, previous/final status, actor type/user id when available, and the redacted audit action name. `POST /api/v1/proposals/{proposalId}/reject` marks a pending or approved proposal rejected without applying its diff, broadcasts `proposal.rejected`, appends a redacted `ai.proposal.rejected` audit row, and requires the same `ai.applyChanges` review permission as approval/application. Rejection audit rows include proposal id, previous/final status, creator type, source id, change count, and changed entity names, but not proposal diffs or provider prompts. Applying a pending, rejected, draft, or already-applied proposal returns `409`, and already-applied or rejected proposals cannot be re-approved for replay. Client-created proposals are always stored as pending review records with `approvalRequired: true`; client-supplied `status`, `approvalRequired`, or `approvedByUserId` values are ignored so proposal review state cannot be forged in the create payload. AI memory facts have structured `type`, optional `subject`, `confidence`, `source`, and a `candidate|approved|rejected|retconned` lifecycle. New user and AI facts are candidates; approval and rejection use their dedicated actions, approved canon must be retconned rather than rejected, and only approved facts enter public/provider canon context. Deletion writes a redacted audit row that records visibility, source count, and lifecycle state without logging memory text.

Scene vision requests return the same permission-filtered polygons used by token visibility checks. Players receive revealed fog polygons, owned-token vision polygons, hide/erase fog polygons, and wall-clipped colored light polygons when fog is active; GM-capable users receive unmasked lighting polygons with `fogActive: false`. Tokens can use the compatibility `visionRadius` field or optional `brightVisionRadius`/`dimVisionRadius` zones; zoned token vision emits separate `lightLevel: "bright" | "dim"` polygons so darkvision-style sight can distinguish close bright visibility from farther dim visibility. Lights can use the compatibility `radius` field or optional `brightRadius`/`dimRadius` zones; zoned lights emit separate `lightLevel: "bright" | "dim"` polygons with dim opacity lower than bright opacity. `GET /api/v1/scenes/{sceneId}/vision/sample?x=...&y=...` returns point-level diagnostics with `visible`, `fogActive`, `revealedBy`, `hiddenBy`, `illuminatedBy`, and `blockedBy` so clients and operators can explain a specific scene coordinate. Sample polygon diagnostics include radius plus token/light level and light color/opacity when applicable, and blocked-wall diagnostics identify the token or light source that was obstructed. `GET /api/v1/scenes/{sceneId}/rendering/diagnostics` requires `scene.update` and returns read-only authoring diagnostics for renderer tooling, including fog/wall/light/source counts, generated polygon vertex counts, source-count complexity summaries, rendering budget posture, issue severity/code rollups, `actionRequired`/`actionReasons`, and warning/error issue codes for invalid fog radius, invalid fog polygons, fog origins or points outside scene bounds, zero-length walls, wall endpoints outside scene bounds, duplicate wall segments, crossing vision-blocking wall segments, duplicate light sources, invalid token vision radius, invalid token vision zone radii, token vision origins outside scene bounds, invalid light radius, invalid light zone radii, invalid light colors, invalid light intensity, and high-complexity vision geometry. Server admins can inspect the same diagnostics across all scenes through `GET /api/v1/admin/rendering/operations`, which returns global scene/fog/wall/light/token-source totals, feature-coverage posture for polygon fog, smoothed fog, terrain walls, colored lights, dimmed lights, bright/dim dual-zone lights, bright/dim token vision, and token vision, a required-feature evidence matrix with per-feature presence, coverage rate, missing feature codes, and bounded scene samples, audited `authoringOperations` for recent fog/wall/light edits with action, target, actor, scene, and campaign rollups, audited `failedAuthoringOperations` for invalid fog/wall/light authoring attempts with action, reason, target, actor, scene, campaign, and bounded recent failure samples, `staleIssueOperations` for flagged scenes with no rendering authoring activity, issue severity/code rollups, max and total polygon vertex budget thresholds, budget usage ratios/exceeded flags, high-complexity geometry totals, scenes requiring action, a bounded remediation queue with affected scenes, sample targets, and the top triggering issue code/severity/message/details for each sample, an audited `actionRequired` flag, and aggregated `actionReasons`. Rendering remediation details preserve safe diagnostic context such as crossing or duplicate wall/light ids, invalid point/intensity values, failed authoring reasons/messages, and exceeded vertex budgets. Scenes that require action but have no fog/wall/light authoring history add a `triage_stale_rendering_issues` remediation row so untouched rendering problems are visible from the operator console. Failed rendering authoring attempts add `rendering_authoring_failures` plus `review_rendering_authoring_failures` remediation without storing map payloads or raw brush geometry. Rendering action reasons are `rendering_errors`, `large_vision_polygon`, `high_vision_vertex_count`, and `rendering_authoring_failures`. Fog authoring accepts circle regions, `shape: "polygon"` with `points`, or `shape: "brush"` with raw freehand `points` and optional `brushRadius`; brush strokes are smoothed and compacted into persisted polygon fog. Use `mode: "reveal" | "hide"` for reveal and erase brushes. Fog regions can be patched with the same validation rules to remediate invalid radius, invalid polygon, or wrong reveal/hide mode diagnostics. Fog create/delete operations append scene-local history entries, `GET /fog/history` exposes the GM-only history log, and `POST /fog/undo` reverses the latest still-applicable create/delete entry. Campaign fog presets save the current fog regions from a source scene, persist in SQLite/archive state, and can be applied to any same-campaign scene through `POST /fog/apply-preset` with `mode: "append" | "replace"`. Wall authoring accepts `kind: "wall" | "terrain"` and optional `blocksMovement`, and scene editors can patch or delete individual wall segments to remediate zero-length, out-of-bounds, duplicate, crossing, or wrong movement-blocking diagnostics. Light authoring accepts `color`, `intensity`, `brightRadius`, and `dimRadius`, and scene editors can patch or delete individual light sources to remediate invalid radius/color/intensity/zone-radii or duplicate-light diagnostics. Token create/update validates `brightVisionRadius` and `dimVisionRadius` so bright token vision must be smaller than dim token vision, and the browser Actor panel exposes selected-token vision enable, dim-radius, and bright-radius controls through that normal token update path. Fog, wall, and light create/update/delete operations are audited and broadcast as scene updates; invalid fog, wall, and light update attempts append redacted failure audit rows before returning validation errors.

Tactical map aids are deliberately advisory. `POST /path-measurement` accepts 2-64 scene coordinates and returns normal distance, distance inside the union of authored difficult-terrain polygons, total distance, and a movement-cost distance that counts difficult segments twice. Square-grid scenes report feet at 5 feet per grid square; gridless scenes report raw scene units. Measurement requires `scene.read` and never moves a token, chooses a route, or enforces wall/collision legality. Difficult-terrain and cover authoring require `scene.update`; all five write routes require `expectedUpdatedAt` plus `Idempotency-Key` and return a structured stale-scene conflict on revision mismatch. Cover is a directional manual ruling for one source-token/target-token pair (`none`, `half`, `three_quarters`, or `total`); it does not infer line of sight or geometry and does not automatically modify Armor Class, attacks, damage, or movement.

The service-provider document advertises strong ETag support. Every provisioning mutation also requires a stable `Idempotency-Key`; successful retries are durably replayed in a credential-scoped SCIM namespace. Uploaded assets are scanned by `builtin-asset-scanner` before storage writes. EICAR test signatures, active SVG content, executable/script uploads, and HTML uploads return `422 asset_security_blocked` with scanner findings and do not create an asset record. When `OTTE_ASSET_TRUST_WEBHOOK_URL` is configured, clean built-in results are also sent to an external AV/trust scanner before quota checks, asset record creation, or local/S3 object writes. The webhook receives JSON with `name`, `mimeType`, `sizeBytes`, `checksum`, and `contentBase64`; optional `OTTE_ASSET_TRUST_WEBHOOK_TOKEN` is sent as a bearer token. Responses use `status: "clean"` or `status: "blocked"` with optional `scanner` and `findings`. Blocked responses, high-severity findings, invalid responses, HTTP failures, or timeouts fail closed by default and return `422 asset_security_blocked`; set `OTTE_ASSET_TRUST_FAIL_CLOSED=false` only when fail-open behavior is intentional. Raster uploads retain their original bytes and may add rebuildable WebP `thumbnail` and `optimized` renditions. Browser media elements should use thumbnail signed URLs for lists and retain the original or optimized URL for the tabletop renderer rather than embedding bearer sessions in query strings. Responses include environment variable names only, never raw values, webhook URLs, or tokens.

When `OTTE_ASSET_STORAGE=s3` or `minio`, `runtime.s3` reports whether the bucket, endpoint, region, force-path-style mode, and explicit credentials are configured. It also flags invalid endpoints, production non-local HTTP endpoints, and partial explicit credentials through delivery configuration warnings without returning bucket names, endpoints, access keys, or secrets.

Scene layer authoring is split into focused endpoints. Fog reveal uses `token.reveal`; wall and light creation use `scene.update` and return the updated scene for realtime rebroadcast.

`POST /api/v1/import/campaign` accepts either a raw `.ottx` archive or:

```json
{
  "archive": {},
  "mode": "upsert"
}
```

The default `upsert` mode replaces records with matching ids and inserts missing records, which supports round-tripping a campaign into a fresh instance or refreshing an existing imported campaign. Use `reject_conflicts` to return `409` when imported ids already exist. Archive imports accept schema versions `0.1.0` and `0.2.0`; new exports are written as `0.2.0`.

The JSON compatibility route includes uploaded asset files in a top-level `files` array, regardless of whether the active backend is local disk or S3-compatible object storage. Each entry stores base64 data plus size and `sha256` checksum metadata. Import validates the asset id, size, and checksum before restoring the file through the configured asset storage provider, so uploaded map backgrounds can round-trip into a fresh instance.

For asset-heavy or near-limit campaigns, use `GET /api/v1/campaigns/{campaignId}/export/stream` and `POST /api/v1/import/campaign/stream` with media type `application/vnd.open-tabletop.ottx-stream`. The additive framed transport carries the same JSON archive metadata followed by raw, length-delimited asset bytes and a binary SHA-256 trailer per asset. Export honors backpressure and never constructs a whole-asset base64 string. Import requires authentication and an idempotency key before body parsing, enforces metadata/per-file/aggregate/file-count caps, stages raw bytes in bounded chunks under the operating-system temporary directory, rejects path-like IDs, duplicate, external, unreferenced, truncated, trailing, size-mismatched, or checksum-mismatched frames, and validates the complete archive before the existing permission/reference/conflict checks and atomic commit. Transport compression is intentionally unsupported: requests with a non-identity `Content-Encoding` return `415`, keeping compressed and expanded limits identical and eliminating compression-ratio ambiguity. Object-storage snapshots remain live until state persistence succeeds, so a failed, cancelled, or disconnected import restores both prior state and prior object bytes and removes normal staging files. `mode`, `scope`, comma-separated `collections`, `regenerateIds`, and `expectedUpdatedAt` are query parameters on the stream import route; an `Idempotency-Key` is mandatory. A non-dry-run stream import that touches an existing campaign must provide that campaign's exact `expectedUpdatedAt`, imports at most one existing campaign per request, and advances the campaign revision after commit. The browser archive workspace uses direct-to-disk saving and File-body upload for this route, reports byte progress, and can cancel while the request remains in flight. Retry restarts from byte zero with the same import idempotency key; byte-range resume is not supported. The JSON routes remain supported for existing `.ottx.json` tooling and small archives.

Every successful non-dry-run archive import also creates an installation-local recovery operation. `GET /api/v1/campaigns/{campaignId}/archive-import-operations` lists bounded summaries, and `GET /api/v1/campaigns/{campaignId}/archive-import-operations/{operationId}/preview` reports the current restore/delete impact plus exact rows whose state or asset bytes have since changed. Both require `campaign.update`. `POST /api/v1/campaigns/{campaignId}/archive-import-operations/{operationId}/rollback` requires the current campaign `expectedUpdatedAt`, an `Idempotency-Key`, and `confirmOperationId` equal to the path id. It applies safe inverse rows and durable pre-import asset bytes off-side, preserves non-overlapping later imports, and returns changed or newly referenced rows as conflicts instead of overwriting them. Recovery rows and their durable object references are operational state and are rejected from portable OTTX payloads.

```bash
curl -fsS \
  -H "Authorization: Bearer $OTTE_SESSION_TOKEN" \
  -o campaign.ottx \
  http://localhost:4000/api/v1/campaigns/camp_demo/export/stream

curl -fsS -X POST \
  -H "Authorization: Bearer $OTTE_SESSION_TOKEN" \
  -H "Idempotency-Key: campaign-import-$(date +%s)" \
  -H "content-type: application/vnd.open-tabletop.ottx-stream" \
  --data-binary @campaign.ottx \
  http://localhost:4000/api/v1/import/campaign/stream
```

Asset storage operations also separate unscanned uploads from managed assets with persisted medium/high trust-scan findings; those assets appear under `asset_trust_warnings` and `review_asset_trust_warnings` with bounded asset ids, scanner names, finding counts, severities, and finding codes but no scanner URLs, tokens, object bytes, or checksums.

`GET /api/v1/campaigns/{campaignId}/dogfood-report-bundle` returns a redacted JSON report for outside dogfood issues. It requires `campaign.read` and includes counts, visibility summaries, scene/token/actor metadata, proposal/eval/tool-call status, content import metadata, and recent audit action ids while omitting emails, auth state, reset/invite tokens, journal and handout bodies, chat bodies, AI messages, AI memory text, AI tool inputs/outputs, raw content-import data, asset URLs, and asset bytes.

Safe content import primitives are separate from campaign archive import. They are campaign-local preview records for user-provided or adapter-normalized content, not a scraper. `POST /api/v1/campaigns/{campaignId}/content-imports/preview` requires `campaign.update` and stores a batch with source adapter metadata, provenance, license usage, per-entity warnings, and default selections. Supported preview entity kinds are `actor`, `item`, `journal`, and `handout`. `POST /api/v1/content-imports/{importId}/apply` selectively creates only the requested records and audits `contentImport.applied`; callers and plugins cannot bypass its typed permission checks or write imported records directly to storage. `POST /api/v1/content-imports/{importId}/rollback` removes records created by that batch and audits the removal. `DELETE /api/v1/content-imports/{importId}` deletes unapplied or rolled-back preview state; applied imports must be rolled back first. D&D Beyond and other external services remain adapter-boundaries only: no scraping, auth bypass, or proprietary payloads should be included, and source/license warnings are stored with the preview.

## D&D inventory, commerce, and typed loot

The strict D&D inventory API stores versioned metadata on normal campaign `Item` records, so actor inventory, party stashes, merchants, and combat loot persist through SQLite and campaign archives without a parallel storage path. Every consequential write requires an `Idempotency-Key` and the current `updatedAt` value for every resource it can change; stale requests return `409` before any mutation.

| Route | Purpose and permission boundary |
| --- | --- |
| `GET /api/v1/campaigns/{campaignId}/dnd/inventory?actorId={actorId}` | Returns a permission-safe actor/stash/merchant/loot overview. Requires `actor.read`; an actor query also requires private access to that actor. Merchant cash is omitted from non-manager responses and other claimants' user ids are redacted. |
| `POST /api/v1/campaigns/{campaignId}/dnd/party-stash` | Creates the one campaign party stash. Requires `campaign.update` and the campaign revision. |
| `POST /api/v1/campaigns/{campaignId}/dnd/merchants` | Creates a merchant with an optional tracked cash balance and bounded catalog. Requires `campaign.update` and the campaign revision. |
| `PATCH /api/v1/campaigns/{campaignId}/dnd/merchants/{merchantId}` | Updates merchant catalog, pricing, buyback rate, or optional liquidity. Requires `campaign.update` plus campaign and merchant revisions. |
| `PATCH /api/v1/campaigns/{campaignId}/dnd/inventory/items/{itemId}` | Updates quantity, weight, parent container, container capacity, extradimensional status, or ammunition link. Requires permission to update the owner plus item and owner revisions. |
| `POST /api/v1/campaigns/{campaignId}/dnd/inventory/items/{itemId}/transfer` | Atomically moves a partial stack or a full container subtree. Party-stash withdrawals require `campaign.update`; every source/destination actor must also be editable by the caller. |
| `POST /api/v1/campaigns/{campaignId}/dnd/inventory/items/{weaponItemId}/consume-ammunition` | Spends a linked, same-owner ammunition stack. Requires actor update permission plus weapon, ammunition, and actor revisions. |
| `POST /api/v1/campaigns/{campaignId}/dnd/merchants/{merchantId}/buy` | Buys catalog stock for an editable actor with copper-exact currency accounting and optional merchant cash tracking. |
| `POST /api/v1/campaigns/{campaignId}/dnd/merchants/{merchantId}/sell` | Sells a non-container actor item at an explicit sell price or merchant buyback rate. Requires actor, merchant, and item revisions. |
| `POST /api/v1/combats/{combatId}/dnd/loot` | Records typed loot as real stash items and links their ids to a combat reward. Requires both `combat.manage` and `campaign.update`. |
| `POST /api/v1/campaigns/{campaignId}/dnd/loot/{itemId}/claim` | Lets a caller claim available loot for an actor they can update. The item remains in the stash until assignment. |
| `POST /api/v1/campaigns/{campaignId}/dnd/loot/{itemId}/assignment` | Assigns claimed loot to its actor with `combat.manage` plus `campaign.update`, or releases a claim by its claimant or a manager. |

Inventory validation rejects missing/foreign parents, non-container parents, container cycles, nesting beyond five levels, and missing/foreign ammunition links. Full containers move with their descendants; partial container transfers and direct container sales are deliberately rejected for manual GM handling. Carrying capacity uses Strength × 15 with D&D size modifiers and supported actor overrides. Container over-capacity is reported rather than silently changing actor rules, and extradimensional contents are excluded from carried weight with an explicit review warning. Currency inputs support D&D denominations and are calculated in copper; canonical balances return GP/SP/CP without floating-point loss. A merchant without `currency` uses manual liquidity and returns a warning while actor funds and stock still update atomically.

The browser exposes this workflow under a D&D actor's Loadout tab. It includes accessible loading/error/retry states, carrying and container warnings, typed quantity/weight/container/ammunition controls, party-stash transfers, merchant buy/sell controls, and claim/assign/release actions for combat loot. `@open-tabletop/api-client` exposes matching typed methods for all twelve routes.

Realtime clients connect to:

```text
ws://localhost:4000/api/v1/realtime?campaignId=camp_demo
```

`@open-tabletop/api-client` includes typed realtime helpers for clients that already use the REST wrapper:

```ts
import type { EngineEvent } from "@open-tabletop/core";
import { OpenTabletopClient } from "@open-tabletop/api-client";

const client = new OpenTabletopClient("https://api.example.test", { token });
const socket = client.connectRealtime("camp_demo");
socket.addEventListener("message", (message) => {
  const event = client.parseRealtimeMessage<EngineEvent>(message);
  console.log(event.type, event.targetId);
});
```

Use `client.connectRealtime(campaignId)` to send the bearer token through the `otte.auth.<token>` websocket subprotocol when a framework or custom WebSocket implementation owns socket construction.

## AI policy, citations, and local privacy

- `GET /api/v1/admin/ai/policy` returns the redacted installation AI policy and production-readiness issues to server admins.
- `GET /api/v1/campaigns/{campaignId}/ai/policy` returns effective installation plus campaign policy to campaign readers.
- `PATCH /api/v1/campaigns/{campaignId}/ai/policy` requires `campaign.update`, `Idempotency-Key`, and `expectedRevision`. The request sets `enabled`, `contextScopes`, `providerTransmissionDisclosure`, and `retentionDays`.
- `POST /api/v1/campaigns/{campaignId}/ai/privacy/preview` requires `campaign.update` and returns exact category counts without prompts, messages, tool inputs/outputs, or other hidden content.
- `POST /api/v1/campaigns/{campaignId}/ai/privacy/prune` requires `campaign.update` and `Idempotency-Key`. It defaults to dry-run; deletion also requires `confirmation: "CLEAR_AI_OPERATIONAL_HISTORY"`. `mode` is `expired` or `all`, `before` bounds time, and `limit` is capped at 1,000 threads.

AI thread responses may include `sources`, validated `citations`, `citationWarnings`, selected `contextScopes`, `policyRevision`, and `retentionExpiresAt`. Unsupported citation claims remain visible with `unknown_source` or `locator_mismatch`; natural-language source-looking text is not parsed as citation metadata.

Privacy pruning removes only local thread, tool-call, and evaluation records selected by the request. Proposals, approved or candidate campaign memory, and aggregate audit records are preserved. `providerDeletion: "not_requested_or_verified"` explicitly separates local deletion from provider-side retention.
