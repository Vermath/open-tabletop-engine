# AI Gateway

The AI layer is provider-agnostic. `packages/ai-core` defines:

- provider interface
- streaming event types
- typed tool definitions
- permission-filtered context builder
- local echo provider for development
- Codex app-server provider bridge

The API endpoint `POST /api/v1/campaigns/{campaignId}/ai/threads` creates a thread and returns an assistant response. `GET /api/v1/campaigns/{campaignId}/ai/threads` gives GMs the campaign's AI thread status history, and `GET /api/v1/campaigns/{campaignId}/ai/usage` aggregates operational usage by provider. Optional evaluation endpoints still exist for deterministic telemetry checks, but they are not part of the active product roadmap and do not drive production actionability. `GET /api/v1/admin/ai/operations` gives server admins a redacted cross-campaign AI/Codex operations summary, including active provider, retry budget, runtime configuration posture, usage, recent threads, recent tool calls, service-level completion/failure rates, duration summaries, per-provider health rollups, proposal-review posture, and an operations risk rollup for failed threads, stale running threads, stale started tool calls, failed tools, provider errors, action-required status, and machine-readable action reasons. The runtime posture flags direct OpenAI API-key provider selections as disabled and points operators to Codex app-server auth before those settings only appear as failed or stale provider work. `POST /api/v1/admin/ai/proposals/stale/reject` lets server admins dry-run or reject stale pending AI proposals with optional campaign scoping, bounded batch size, and redacted proposal samples; `includeApproved: true` extends the cleanup to stale approved-but-unapplied AI proposals without applying their diffs. `POST /api/v1/admin/ai/threads/stale/fail` lets server admins dry-run or mark stale running threads as failed with optional campaign scoping, bounded batch size, and redacted audit evidence. `POST /api/v1/admin/ai/tool-calls/stale/fail` provides the same dry-run and cleanup path for stale started tool calls, optionally scoped by campaign or thread. `POST /api/v1/admin/ai/tool-calls/retry` lets server admins dry-run or manually replay retryable failed tool calls by campaign, thread, or tool-call id; replay resolves the original started-call input, rechecks the original thread user's current campaign permissions, and records a fresh started/completed or started/failed lifecycle pair instead of silently mutating the old failed record. These admin operations append redacted audit logs with aggregate counts and cleanup, reject, or retry outcomes, not prompts or tool inputs. The web AI panel shows GM-only operational rollups, recent threads, provider usage, recent tool calls, and normal approve/apply or reject controls for reviewable proposals; the web Admin tab shows the server-admin AI operations summary, runtime configuration posture, runtime failure-rate and p95 duration metrics, per-provider failure/completion/cost/stale-running posture, pending/apply-ready proposal review pressure, applied and rejected proposal recency, stale proposal counts and samples, failed proposal apply counts and samples, recent thread status/provider/duration rows, stale cleanup counts and controls, separate stale pending and stale approved proposal rejection controls, recent stale record rows with provider and age, top redacted provider and failed-tool rollups, per-tool retryable versus non-retryable failure policy, recent tool-call error codes, retryable failed-tool count, and a retry control for eligible recent failed tool calls. The web client wraps assistant output in a reviewable proposal before applying campaign changes. Proposal review state is server-owned: create payloads cannot forge `approved`/`applied` state or bypass `approvalRequired`, apply requires an approved proposal, reviewers can reject pending or approved proposals without applying their diffs, normal rejection appends redacted `ai.proposal.rejected` audit evidence without proposal diffs or prompts, and already-applied or rejected proposals cannot be re-approved for replay. Server-admin AI operations now group pending and approved-but-unapplied AI proposals by campaign and changed entity, expose bounded redacted recent applied and rejected proposal samples, flag pending or approved queues older than 24 hours, roll failed apply attempts into redacted `ai_proposal_apply_failures` posture, and escalate stale or failed proposal-review remediation without exposing full proposal diffs.

Provider packages include the local echo provider and `@open-tabletop/codex-app-server-provider`, which defines the Codex App Server JSON-RPC transport bridge, maps Codex events into OpenTabletop AI provider events, and consumes Codex `imageGeneration` items for map and token asset generation.

Configure `OTTE_AI_PROVIDER=codex-loopback` to keep using the deterministic offline Codex adapter for smoke tests. Configure `OTTE_AI_PROVIDER=codex-app-server` to connect the API to a live local Codex app-server over WebSocket. By default, a local loopback listener such as `ws://127.0.0.1:4500` is started on first use with the local Codex launcher; set `OTTE_CODEX_APP_SERVER_AUTOSTART=false` to require a separately managed listener, or set `OTTE_CODEX_APP_SERVER_COMMAND` only when the Codex npm shim is installed somewhere custom. On Windows, the default auto-start path prefers the local `codex.cmd` npm launcher instead of the PowerShell shim. On Railway, the API service bundles the Codex CLI and starts that bundled launcher in-container. Set `OTTE_CODEX_APP_SERVER_LOGIN_TYPE=chatgptDeviceCode` for hosted deployments where the browser cannot reach a localhost OAuth callback. Optional Codex settings are `OTTE_CODEX_APP_SERVER_CWD`, `OTTE_CODEX_MODEL`, `OTTE_CODEX_IMAGE_MODEL`, and `OTTE_CODEX_MODEL_PROVIDER`. Direct OpenAI API-key runtime settings are intentionally not supported by the app; OpenAI-backed model and image work is routed through Codex app-server's authenticated account. The live transport follows the official Codex app-server JSON-RPC flow with experimental dynamic tools enabled: OpenTabletop advertises only the caller-visible permission-filtered tools, Codex can request map and token generation through `open_tabletop.generate_map_asset` and `open_tabletop.generate_token_asset`, and the API still executes those requests through the normal proposal-backed tool handlers. The bridge uses read-only sandbox settings, denies Codex shell/file approval requests, and keeps campaign mutation behind OpenTabletop permissions and proposal review. Official setup details live in the OpenAI Codex docs for [app servers](https://developers.openai.com/codex/app-server) and [remote app-server connections](https://developers.openai.com/codex/cli/features#connect-the-tui-to-a-remote-app-server).

Provider calls retry once before any event is emitted by default; set `OTTE_AI_PROVIDER_RETRY_ATTEMPTS` from `0` to `3` to adjust that pre-event retry budget. App-server requests use `OTTE_AI_PROVIDER_TIMEOUT_MS`, defaulting to `30000`; timeout and retry controls emit `ai_runtime_control_config_invalid` when configured with non-numeric or negative values. Set `OTTE_AI_PROVIDER_FAILURE_RATE_THRESHOLD` from `0` to `1` to tune when per-provider health marks a provider degraded by failure rate; the default is `0.25` after at least three provider threads. Set `OTTE_AI_PROVIDER_P95_DURATION_THRESHOLD_MS` to mark provider health degraded when p95 thread duration exceeds a configured millisecond threshold after at least three timed provider threads. Set `OTTE_AI_PROVIDER_RUNNING_THREAD_THRESHOLD` to mark provider health under running-thread pressure before active work ages into stale failures; the default is `10`. Non-numeric or negative provider health thresholds emit `ai_provider_threshold_config_invalid` with the affected environment variable names but not raw values. Set both `OTTE_AI_INPUT_TOKEN_COST_USD_PER_1K` and `OTTE_AI_OUTPUT_TOKEN_COST_USD_PER_1K` to calculate estimated costs from provider-reported token usage; configuring only one side emits `ai_cost_rates_partial` in server-admin runtime posture because estimated costs are intentionally not calculated from a partial rate pair. Non-numeric or negative AI cost settings emit `ai_cost_config_invalid` with the affected environment variable names but not raw values. Set `OTTE_AI_COST_BUDGET_USD` to expose configured budget, remaining spend, usage ratio, and `cost_budget_exceeded` actionability in server-admin AI operations.

Hosted-model instructions explicitly remind the provider that campaign edit tools create reviewable proposals and that it must not claim scenes, actors, journals, tokens, encounters, or memory changed until tool output confirms approval or creation. The instructions also require exact visible actor action ids for actor-action tools, prohibit filling missing rules, compendium, campaign, or character details from outside knowledge, and tell the model to use visible compendium tools or ask for clarification instead of guessing hidden or unavailable actors, targets, scenes, actions, or rules.

AI thread tools currently include:

- `create_proposal` for generic GM-approved campaign changes.
- `draft_encounter` for encounter proposal drafts.
- `draft_journal_entry` for journal-entry proposal drafts.
- `draft_scene` for scene proposal drafts.
- `draft_token_update` for token update proposal drafts.
- `generate_map_asset` for generated map image assets, staged as asset-create proposals and optional scene background updates.
- `generate_token_asset` for generated token art assets, staged as asset-create proposals and optional token `imageAssetId` updates.
- `draft_actor_update` for actor update proposal drafts.
- `create_memory` for queued memory facts.
- `search_memory` for approved memory facts visible to the caller, with GM-only scopes denied unless the caller has GM memory permission.
- `read_chat`, `read_roll`, `read_journal`, `read_scene`, `read_token`, `read_asset`, `read_combat`, `read_encounter`, and `read_actor` for permission-safe campaign context reads through normal chat, journal, scene, token, combat, campaign, and actor permissions.
- `roll_dice` for campaign dice rolls posted to chat.
- `use_actor_action` for permission-checked actor action rolls, optional `spellSlotLevel` upcasting, resource consumption, and supported target damage/healing application.
- `read_compendium` for permission-safe `dnd-5e-srd` rules compendium lookups, with legacy demo systems still available.

Image generation uses Codex app-server only. `OTTE_AI_IMAGE_PROVIDER` may be left unset when `OTTE_AI_PROVIDER=codex-app-server`, or explicitly set to `codex-app-server`; API-key providers and `local-svg` placeholders are not runtime options. `OTTE_CODEX_IMAGE_MODEL` can override the image-generation turn model, otherwise `OTTE_CODEX_MODEL` or the Codex default is used. The app-server schema exposes generated images as `imageGeneration` items with base64 `result` bytes, an optional revised prompt, and an optional Codex `savedPath`; this app consumes those bytes directly and stores a raster asset. The browser can call `POST /api/v1/campaigns/{campaignId}/ai/generate-map-asset` and `POST /api/v1/campaigns/{campaignId}/ai/generate-token-asset`; Codex providers can call the matching tools in an AI thread. Both paths execute the same permission-checked tool handlers. Generated asset bytes are written to configured asset storage immediately, but the asset record and scene/token attachment remain pending proposal changes until a permitted user applies them. If Codex app-server is unavailable, lacks image-generation capability, or returns no image result, the tool fails visibly rather than substituting an SVG fallback.

Every tool is checked against the human caller's campaign permissions before execution. The provider only receives tool definitions whose declared permissions are already available to the caller, and execution-time checks still enforce the same requirements if a stale or malicious provider emits an unavailable tool call. Provider-visible tools fail closed: only the allowlisted permission-safe tools `search_memory`, `read_chat`, `read_roll`, `read_journal`, `read_scene`, `read_token`, `read_asset`, `read_combat`, `read_encounter`, `read_actor`, `roll_dice`, and `read_compendium` may omit `ai.proposeChanges`; every other current or future tool must require `ai.proposeChanges` before advertisement or execution. Proposal-backed campaign edit tools also require the underlying edit permission, and the generic `create_proposal` tool rejects proposal changes whose underlying edit permission is missing. Tool inputs are validated against each tool's required schema before side effects run, so malformed provider function-call arguments are returned as failed tool outputs instead of default-valued campaign mutations. Started, completed, and failed tool calls are persisted with completion durations, and GMs can inspect them through `GET /api/v1/campaigns/{campaignId}/ai/tool-calls`. Server-admin operations classify failed tool calls by retry policy and support manual replay for retryable failures only: `tool_failed` and `stale_tool_call` are retryable, while permission, schema, and unknown-tool failures remain non-retryable until configuration or provider output changes. Threads persist operational status fields including `running`, `completed`, or `failed`, start/end timestamps, duration, retry attempts, event count, tool-call count, advertised provider tool names, advertised provider tool required permissions, server-owned `permissionSafe` classification, assistant response text, prompt/context/response character counts, provider token usage, estimated cost when rates are configured, and provider error text when a provider call fails. Current AI production depth work is focused on provider reliability, permission-safe tool execution, proposal review, usage/cost visibility, and runtime operations posture rather than model-output evaluation programs.
